# Design Notes

## Why rule-based over ML?

법학 전공자로서 탐지 로직이 **설명 가능(explainable)** 해야 실무에서 의미 있다고 판단했습니다. 머신러닝 모델은 false positive를 줄이는 데 유리하지만, 각 alert가 왜 발생했는지 법적 근거로 설명하기 어렵습니다. 본 데모는 "규정을 쿼리로 번역"하는 학습이 주목적이므로 rule-based만 다뤘습니다. 실무에서는 둘의 병행이 표준입니다.

## Why synthetic data?

실거래 데이터는 개인정보보호법 및 금융실명법상 외부 사용이 금지됩니다. 또한 학습 목적에서는 오히려 **의도된 의심 시나리오를 심어둘 수 있는** 가상 데이터가 유리합니다. `data/generate_synthetic_data.py`에서 7개의 시나리오를 명시적으로 주석 처리해 둔 것도 이 때문입니다.

## Why Claude API, not LangChain?

Module 2의 경우 LangChain의 추상화 레이어가 프로토타입 단계에서 디버깅을 어렵게 합니다. Claude API 직접 호출이 요청/응답 흐름을 단순하게 유지해 요약 결과가 이상할 때 원인 추적이 수월합니다. 법학 전공자가 "AI가 무엇을 입력받아 어떤 판단을 했는지" 직접 검증하기 위한 선택이기도 합니다.

## Why mock mode?

외부 API 키·네트워크가 없는 환경(서류 심사자, 면접관 로컬 등)에서도 레포를 즉시 실행해볼 수 있도록 mock mode를 기본 지원합니다. `python src/main.py --mock` 한 줄로 `sample_outputs/`의 리포트가 재생성됩니다. 실무에서도 테스트·데모 환경 분리는 표준 관행입니다.

## Why human-in-the-loop?

특금법상 내부통제 책임은 금융기관과 담당 임직원에게 귀속됩니다. AI 자동화는 담당자의 인지 부하를 줄일 수 있으나, 최종 보고 판단을 AI가 대신할 수 없다는 것이 본 설계의 출발점입니다. Module 2의 출력물이 "담당자가 30초 스캔 후 최종 판단"을 전제로 설계된 이유입니다.

## Tradeoffs

| 결정 | 선택 | 근거 |
|---|---|---|
| 수집 주기 | 일 배치 | 담당자 업무 리듬 및 API 비용 고려. 실시간이 반드시 필요한 공지는 드묾 |
| 요약 길이 | 3~5문장 | "30초 스캔 가능"을 기준으로 삼아 길이 상한을 정함 |
| 모델 선택 | Sonnet | 한국어 금융 전문용어 번역 정확도 우선. Haiku는 속도는 빠르나 용어 오역 가능성 존재 |
| 모듈 분리 | 독립 모듈 2개 | 각 모듈이 독자적 목적을 갖도록. 통합 시 오히려 설계 의도가 흐려짐 |
| 룰 위험 가중치 | 수동 heuristic | 통계적 튜닝을 주장하려면 실데이터가 있어야 하므로, 본 데모에서는 명시적 heuristic으로 한정 |

## 한계 인식

본 프로젝트는 학습 목적 프로토타입이며, 실무 시스템에서 요구되는 다음 요소는 포함되지 않습니다.

- 실데이터 기반 룰 calibration 및 backtesting
- 머신러닝 기반 FP 감소 파이프라인
- 내부 통제 감사 로그, 보고 워크플로우, 담당자 권한 관리
- 시스템 연속성(장애 복구, 큐잉, 재처리)
- 개인정보 보호를 위한 접근 통제·암호화

이러한 요소는 실무 투입 시 반드시 별도 검토가 필요합니다.
